## 1. 项目基线与边界

当前项目是 Vue 3 + TypeScript + Vite 单页应用：

- `src/domain/machine.ts`：纯有限状态机，负责阶段、任务队列、结果应用和终局。
- `src/domain/engine.ts`：条件过滤与确定性加权抽取。
- `src/domain/catalog.ts`：273 个转盘、25 个标签、2,456 个事件的只读目录。
- `src/composables/useGameStore.ts`：动画、自动推进、回退、本地存档和 UI 编排。
- `src/components/FateWheel.vue`：Canvas 盘面与旋转动画。
- `src/data/wheels.json`：内置源数据，不在浏览器运行时直接改写。
- `tests/`：Vitest 数据完整性和状态迁移回归。

必须保持的边界：

- 业务阶段只由状态机迁移改变。
- 相同种子、相同有效转盘配置和相同操作顺序得到相同结果。
- UI 历史与可编辑转盘覆盖是不同状态域；返回上一转盘不隐式回退全局配置修改。
- 覆盖层按稳定池 ID 合并，不污染角色存档，也不直接变异原始 JSON。
- 盘面角度、编辑器占比、抽取概率和选中落点来自同一份有效候选项。
- PC 与移动端使用同一份业务数据和操作，不建立平行游戏逻辑。

## 2. 计划清单

更新迭代计划时检查：

- 版本、日期、状态和范围。
- 现状问题、目标、明确不做事项。
- PC、平板、移动端布局与交互差异。
- 受影响组件、composable、领域模块、数据格式、存储键和下载文件。
- 状态机、回退、自动推进和编辑配置之间的并发边界。
- 权重缺省、归一化、条件过滤、零候选和极端权重策略。
- localStorage 版本、损坏数据恢复、默认值恢复和跨版本兼容策略。
- 分阶段实施顺序、依赖、优先级、估时和回滚策略。
- 单元、组件、真实浏览器、视觉、可访问性和下载验证。
- 每项验收标准的稳定 ID 和验证方式。

## 3. 功能与风险覆盖地图

| 功能域 | P0 行为 | 主要验证 |
|---|---|---|
| 新命运 | 随机、人类、魂兽路线可启动；种子可重现 | 状态机 + 浏览器 |
| 手动推进 | 任务、候选、动画、结果、概率、日志同步 | Vitest + Playwright |
| 自动/极速 | 开始、暂停、终局停止、无并发投掷 | store 测试 + Playwright |
| 返回上一转盘 | 原子恢复角色、队列、日志、RNG 与盘面；连续返回 | Vitest + Playwright |
| 加权转盘 | 角度、标签、落点和实际概率一致 | 纯函数 + Canvas 像素/浏览器 |
| 当前转盘编辑 | 权重、启停、新增、验证、取消、应用和恢复默认 | 组件 + Playwright |
| 覆盖持久化 | 刷新后生效、损坏数据降级、与角色存档隔离 | Vitest + Playwright |
| 修改导出 | 菜单状态、文件名、MIME、schema、池摘要和最终事件 | Vitest + 浏览器下载 |
| 角色信息 | 人类/魂兽/化形字段、空值、长标签 | 组件 + 多视口截图 |
| 命运纪事 | 搜索、重大筛选、倒序、导出传记 | 组件 + Playwright |
| 存档 | 保存、读取、JSON 导入导出、rolling 恢复 | Vitest + 浏览器下载 |
| 移动导航 | 主舞台/角色/纪事、更多菜单、固定操作栏 | 多视口 Playwright |
| 开始与终局 | 路线弹窗、取消、死亡/存活终局、再来一局 | 状态机 + Playwright |
| 高级工具 | 自由转盘不污染主流程；设置持久化 | Vitest + Playwright |
| 可访问性 | 键盘、焦点、文本状态、减弱动画、44px 点击区 | Playwright + 人工审计 |
| 数据完整性 | 273 池、25 标签、2,456 事件不丢失 | Vitest |

每次迭代只执行范围内的 P2，但必须覆盖受影响功能的 P0/P1 回归。

## 4. 测试用例 JSON

默认文件：`docs/test/<version>-test-cases.json`。

```json
{
  "suite": "douluo-spin-v0.1",
  "plan": "docs/iteration-plan-v0.1.md",
  "description": "本套件验证的用户结果。",
  "environment": {
    "app": "运行时注入的本地 Vite URL",
    "storage": "隔离浏览器上下文"
  },
  "cases": [
    {
      "id": "V01-WHEEL-01",
      "requirementIds": ["V01-WEIGHT-01"],
      "title": "盘面按实际权重显示并落入正确扇区",
      "phase": "Phase B",
      "category": "加权转盘",
      "priority": "P0",
      "precondition": "当前池包含权重 1:3:6 的三个可抽事件。",
      "steps": ["启动固定种子命运", "打开当前转盘", "转动并等待结算"],
      "expected": "扇区为 10%:30%:60%，指针落入抽中事件扇区，结果概率一致。",
      "verification": ["unit", "browser", "canvas-pixels"],
      "evidence": ["screenshot", "trace"]
    }
  ]
}
```

规则：

- `id` 在后续修订中保持稳定。
- `requirementIds` 必须能回到计划验收项。
- `steps` 写用户动作，不写实现细节。
- `expected` 包含可观察 UI、状态、存储或下载结果。
- 浏览器用例不得只写“页面正常”；明确元素、值、状态变化和禁止行为。
- 修复 bug 时新增回归用例，不覆盖掉原失败事实。

## 5. 双端浏览器矩阵

至少验证：

| 类型 | 视口 | 重点 |
|---|---:|---|
| 最小移动端 | 320x568 | 文本换行、编辑器、底栏遮挡 |
| 常见移动端 | 375x667 | 首屏闭环、转盘 <= 280px、标签切换 |
| 长屏移动端 | 390x844 | 纪事滚动、更多菜单、下载 |
| 平板竖屏 | 768x1024 | 双栏/单栏切换、弹窗尺寸 |
| 平板横屏 | 1024x768 | 转盘与结果并排、纪事跨栏 |
| 常见桌面 | 1366x768 | 首屏信息密度、独立滚动 |
| 宽桌面 | 1440x900 | 转盘 <= 420px、侧栏布局 |

每个视口检查：

- 无页面级横向滚动。
- 文本、按钮、菜单、弹窗、固定底栏和终局横幅不重叠。
- 点击区、焦点和键盘顺序合理。
- Canvas 非空、尺寸稳定、没有被裁切或拉伸。
- 主内容底部留白覆盖安全区与固定操作栏。

## 6. Playwright 验证规则

- 优先使用 role、label 和稳定测试 ID；不要依赖易变的深层 CSS 路径。
- 为每个测试创建隔离 context；需要验证刷新持久化时只在同一 context 内复用 storage。
- 使用固定种子准备确定性流程。需要特定阶段时优先通过受支持的存档导入或应用操作准备，不直接篡改 Vue 内部状态。
- 等待可见状态、动画结束或下载事件，不用长固定 sleep 掩盖竞态。
- 对自动推进使用明确的开始、暂停和步数断言。
- 使用 `page.waitForEvent('download')` 验证文件名、内容类型和 JSON schema。
- 读取 Canvas 像素或截图裁剪区，验证存在足够多的非背景像素；权重用例配合纯函数角度断言。
- 对 `prefers-reduced-motion: reduce` 建立独立 context，断言没有长旋转等待。
- 失败时保留 screenshot、video、trace 和控制台错误；控制台未捕获异常视为失败。
- 测试完成后确认没有遗留 Vite、Playwright 或录制进程。

建议脚本名（不存在时才新增）：

```json
{
  "test:e2e": "playwright test",
  "test:e2e:headed": "playwright test --headed",
  "test:verify": "npm test && npm run build && playwright test"
}
```

不要仅为匹配建议名覆盖仓库已有等价脚本。

## 7. 录屏与证据

建议证据根目录：`test-results/iterations/<version>/`。

```text
test-results/iterations/<version>/
  desktop/
    screenshots/
    video/
    trace.zip
  mobile/
    screenshots/
    video/
    trace.zip
  downloads/
  reports/
    browser-results.json
    coverage-matrix.md
```

录屏流程至少包含：

1. 打开真实应用并创建固定种子命运。
2. 查看当前任务和按权重绘制的转盘。
3. 完成一次转动并看到结果、概率与纪事更新。
4. 返回上一转盘并验证状态恢复。
5. 范围包含编辑器时，修改权重、增加事件、应用并再次转动。
6. 从右上角菜单导出修改 JSON，并验证下载内容。
7. 移动端切换主舞台、角色和纪事，操作固定底栏及更多菜单。

证据默认属于测试产物，不要把大视频提交到源码目录；如果用户要求纳入仓库，先确认目标目录和体积策略。

## 8. 覆盖审计与文档回填

生成矩阵列：

| 计划验收 ID | 测试 ID | 自动化文件 | 最新结果 | 证据 | 缺口/动作 |
|---|---|---|---|---|---|

审计顺序：

1. 从计划提取全部 P0/P1 验收项。
2. 从测试 JSON 提取 `requirementIds`。
3. 从 Vitest 和 Playwright 提取实际测试名称。
4. 对照最新报告和证据目录，不把“写了测试”当作“测试通过”。
5. 补齐缺失覆盖或记录明确排除理由。
6. 发生修复后重新运行相邻主流程和完整基准。

文档回填：

- 状态、事件或迁移变化：更新 `docs/state-machine.md`。
- 技术栈、命令、目录或核心能力变化：更新 `README.md`。
- 范围、阶段或验收事实变化：更新对应 `docs/iteration-plan-*.md`。
- 新增自动化：创建或更新 `docs/test/<version>-automation.md`。
- 存储键、导出 schema 或覆盖层变化：在最贴近的权威文档中写清当前格式与兼容规则。

自动化文档至少记录范围、来源计划、环境、命令、测试清单、覆盖缺口、浏览器矩阵、证据路径、修复记录和剩余风险。
