# 斗罗转盘全流程迭代

## 概览

驱动一次可追踪、可复现、有浏览器证据的 douluoSpin 完整迭代。将计划、测试、实现、双端验证、录屏、覆盖审计和文档回填视为同一个交付，不在代码刚完成时提前结束。

不要把 AgentChat 参考项目的命令、端口或架构复制到本项目。始终先读取当前仓库，以真实文件和命令为准。

## 必读来源

开始工作前读取：

- 用户指定的迭代计划；未指定时优先查找 `docs/iteration-plan-*.md`，在多个候选中选择最新且与请求匹配的文件。
- `README.md`、`docs/state-machine.md`、`package.json`、`src/domain/types.ts`、`src/domain/catalog.ts`、`src/domain/engine.ts`、`src/domain/machine.ts`。
- `tests/`、已有端到端目录、测试配置和与本轮范围直接相关的组件、store、样式及数据文件。
- [references/douluo-spin-workflow.md](references/douluo-spin-workflow.md)：完整执行时读取全部；只制定计划时读取“计划清单”和“项目边界”。

保留用户指定文档的结构并原地更新，不创建内容重复的平行计划。

## 判断请求模式

按用户动词确定执行深度：

- 用户要求“制定/设计全流程计划”：只完成范围分析、计划和用户明确要求的测试设计；不要创建 goal 或修改业务代码。
- 用户要求“执行/开始/完成/实现全流程迭代”：执行下述完整工作流；goal 工具可用时必须用 goal 跟踪。
- 用户要求“继续/恢复完整迭代”：读取已有 goal、计划、测试结果和工作区现状，从第一个未完成门禁继续，不重做已验证工作。

## 完整工作流

### 1. 稳定范围

- 读取计划、状态机、现有实现和测试，确认目标版本、范围、不做事项、受影响组件、领域模块、存储键、数据格式和验证命令。
- 将计划中的每条验收标准映射到实现位置和验证方式。
- 明确工作区已有改动并保留它们；不要回滚无关修改。
- 发现计划与代码事实冲突时，先修订计划并记录决策，再开始实现。

### 2. 更新迭代计划

- 确保计划包含背景、现状、目标、不做事项、双端信息架构、状态与数据边界、分阶段任务、风险、验收标准和工期。
- 为状态机、随机确定性、转盘权重、回退、本地覆盖、导入导出、移动端和可访问性建立覆盖矩阵。
- 给阶段和验收项分配稳定 ID，供测试用例引用。
- 对新增范围同步调整实施顺序和估时，不保留互相冲突的旧表述。

### 3. 生成结构化测试用例

- 默认写入 `docs/test/<version>-test-cases.json`；目录不存在时创建。
- 使用稳定 ID、计划阶段、分类、优先级、前置条件、步骤、预期、验证层和证据要求。
- 先覆盖 P0 主流程，再覆盖状态机边界、错误状态、持久化、导入导出、响应式、键盘操作、减弱动画和回归风险。
- 每个 bug 修复都增加回归用例；每条计划验收标准至少对应一个测试 ID。
- 使用 reference 中的 JSON 形状，沿用已有更严格 schema 时不要降级。

### 4. 启动 goal 跟踪

- 仅在用户要求执行完整工作流且 goal 工具可用时创建 goal。
- 目标格式使用：“实现 `<计划路径>` 的 douluoSpin 迭代，通过 `<测试用例路径>`、项目测试和真实 Playwright 双端验证，并完成录屏、覆盖审计与文档回填。”
- 维护可见阶段清单：范围/计划、测试设计、实现、单元验证、桌面浏览器、移动浏览器、录屏证据、遗漏审计、文档回填、最终回归。
- goal 工具不可用时使用计划工具跟踪相同门禁，并在最终说明该限制。
- 在所有门禁完成前不要将 goal 标记为完成。

### 5. 按项目边界实施

- 遵循 Vue 3 Composition API、TypeScript、纯领域状态机和现有组件模式。
- 让 `transition()` 继续成为业务阶段变化的唯一入口；Vue store 只负责编排动画、历史、本地存储和 UI 状态。
- 保持相同种子、相同配置和相同操作顺序的结果确定性。
- 不直接变异导入的 `wheels.json`；可编辑转盘使用按池 ID 合并的独立覆盖层。
- 让抽取、概率显示、Canvas 扇区和落点复用同一份有效候选项与归一化权重。
- 复用 PC/移动端业务组件，通过布局和明确的展示变体适配，避免复制状态和行为。
- 每完成一个阶段就运行最小相关验证，发现问题立即修复，不把已知失败推到收尾。

### 6. 运行代码级验证

- 以 `package.json` 当前脚本为准，至少运行 `npm test` 和 `npm run build`。
- 为纯函数、状态迁移、权重归一化、覆盖合并、导出结构和确定性优先写 Vitest。
- 数据完整性回归必须保留：273 个转盘、25 个标签、2,456 个选项，除非计划明确包含数据迁移且文档同步更新。
- 测试失败时区分实现问题、用例问题和环境问题；修正后重新运行受影响集合及完整基准。

### 7. 用真实浏览器验证双端

- 优先复用现有 Playwright 配置和脚本；不存在时为本轮范围建立最小 `@playwright/test` 验证能力，并在 `package.json` 中加入明确命令。
- 启动真实 Vite 服务，记录实际 URL；端口被占用时使用另一个端口，不终止无关服务。
- 使用 reference 中的视口矩阵验证桌面、平板和移动端。测试真实 Canvas、localStorage、下载、菜单、弹窗、固定底栏、标签切换和长文本。
- 不用截图代替断言。先断言用户可见状态和持久化结果，再保存截图、视频或 trace 作为证据。
- 对 Canvas 做非空像素检查；权重相关范围还要断言扇区角度或落点与归一化权重一致。
- 失败时保存截图和 trace，修复后重新运行该用例及相邻 P0 流程。

### 8. 录制真实用户流程

- 代码级与浏览器测试通过后录制至少一条桌面和一条移动端完整流程；使用 Playwright video 或等价的真实浏览器录制，不用静态拼图冒充操作录像。
- 覆盖功能入口、核心操作、状态变化、结果反馈和持久化/导出结果。本轮涉及转盘编辑时必须演示修改权重、增加事件、应用、转动和导出 JSON。
- 在关键操作前后保留足够画面停留；关闭会暴露本地路径、密钥或无关个人信息的 UI。
- 将视频、截图、trace 和报告放在统一迭代证据目录，并在自动化测试文档中记录。
- 用户明确跳过录屏时可以跳过，但必须在最终结果和未完成门禁中说明。

### 9. 审计遗漏并回填文档

- 实现和测试完成后重新读取计划、`README.md` 和 `docs/state-machine.md`。
- 按“计划验收项 -> 测试 ID -> 自动化文件 -> 最新结果 -> 证据”生成覆盖表；补齐未覆盖的 P0/P1，或写明排除原因。
- 新增或改变状态、事件、数据格式、存储键、组件职责、运行命令、用户流程或验证方式时，更新对应项目文档。
- 将文档写成当前事实，不只记录迭代过程；删除已经过期或冲突的旧描述。
- 默认写入 `docs/test/<version>-automation.md`，记录命令、覆盖、证据、失败修复和剩余风险。
- 文档更新后再运行必要验证，确保文档中的命令和路径真实存在。

### 10. 带证据收尾

- 确认工作会话中的服务、测试和录制进程都已结束；供用户试用的开发服务器除外，并给出 URL。
- 汇总计划、测试 JSON、自动化文档、核心实现文件和证据目录。
- 报告每条验证命令的结果、测试数量或关键断言、浏览器视口和未解决风险。
- 明确说明哪些项目文档已回填；未更新时说明为什么本轮没有改变对应事实。
- 只有在实现、测试、双端浏览器、录屏或明确豁免、覆盖审计、文档回填和最终回归全部完成后，才将 goal 标记为完成。

## 强制停止条件

仅在以下情况停止并请求用户输入：

- 必须使用用户未授权的生产服务、账号、密钥或破坏性操作。
- 计划存在会实质改变产品语义的互斥选择，且仓库无法推断用户意图。
- 外部依赖或系统权限在安全替代方案耗尽后仍阻止关键验收。

本地类型、构建、测试、浏览器、Canvas、选择器、布局或实现错误不是停止理由；继续诊断并修复。

## 完成门禁

不要在以下任一项未满足时声称完整迭代已完成：

- 计划与实际范围一致，且没有遗留冲突描述。
- 结构化测试用例覆盖全部 P0/P1 验收项。
- 单元测试、类型检查和生产构建通过。
- PC 与移动端真实浏览器主流程通过，无横向溢出、遮挡或空白 Canvas。
- 关键下载和本地持久化经过真实验证。
- 录屏证据存在，或用户明确同意跳过。
- 覆盖缺口已审计，发现的实现问题已修复并回归。
- 项目文档已按当前事实更新。
