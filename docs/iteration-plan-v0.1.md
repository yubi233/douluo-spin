# 迭代计划 v0.1：双端界面重构

## 1. 迭代目标

本轮不改写内置转盘默认数据、状态机迁移、角色存档格式和剧情数据；用户调整通过独立的本地覆盖层改变运行时有效事件与权重。

核心目标：

1. 缩小转盘占用，让当前角色、当前任务、本次结果和最近经历在首屏获得更高优先级。
2. 保留 PC 端高信息密度，同时为移动端建立独立的信息架构，而不是简单纵向堆叠桌面面板。
3. 确保“查看当前状态 -> 转动 -> 阅读结果 -> 继续推进”这一主循环在双端都能单手或单屏完成。
4. 支持返回上一转盘，并完整恢复该步开始前的角色、任务、结果、日志、随机数与盘面状态。
5. 让盘面扇区真实反映事件权重，并允许用户编辑当前转盘的事件、权重与启用状态。
6. 支持导出所有本地转盘修改，形成可交给开发者审阅和合并的 JSON。
7. 保持现有深色、金色与青绿色视觉语言，不照搬参考图的浅色皮肤。

## 2. 当前问题

| 问题 | 当前表现 | v0.1 处理方向 |
|---|---|---|
| 转盘权重过高 | 最大 510px，移动端可达容器 96%，挤压结果与角色信息 | PC 限制在 360-420px，移动端限制在 240-300px |
| 移动端信息链过长 | 转盘、角色、纪事依次纵向排列 | 首屏保留摘要、任务、结果和转盘；详情通过标签页切换 |
| 角色信息离操作区较远 | PC 位于独立栏，移动端位于转盘面板之后 | 增加紧凑角色摘要条，完整详情继续由角色面板承载 |
| 最近剧情反馈不够突出 | 结果有展示，但最近纪事需要进入第三栏或向下滚动 | 主舞台增加“最近经历”摘要，完整纪事仍可搜索和筛选 |
| 权重没有体现在盘面 | 抽取逻辑读取 `weight`，Canvas 仍将所有扇区等分 | 按当前有效权重计算扇区角度、标签位置和最终落点 |
| 无法在运行时调整事件 | 原始 `wheels.json` 作为只读目录直接加载 | 增加独立的本地覆盖层，由当前转盘编辑器写入 |
| 撤销后盘面可能不一致 | 状态机快照可以回退，但 `wheelOptions`、选中项和动画角度是独立 UI 状态 | 回退时从恢复后的活动任务重建盘面并清除后一步动画状态 |
| 移动端顶栏拥挤 | 状态、撤销、新命运等操作共同占据两行 | 顶栏只保留返回、品牌、状态和更多菜单；推进操作固定在底部 |
| 响应式只有单一断点 | 760px 以下直接改为单列 | 定义移动、平板、桌面、宽屏四档布局行为 |

## 3. 信息优先级

页面内容按以下顺序组织：

1. 当前是谁：姓名、路线、年龄、性别、等级或修为、存活状态。
2. 当前要做什么：阶段、任务标题、候选结果数量。
3. 刚刚发生什么：本次结果、概率、投掷次数。
4. 下一步操作：返回上一转盘、继续剧情、自动推进、极速结算。
5. 成长详情：武魂、血脉、魂环、魂骨、天赋、领域、阵营与时间坐标。
6. 历史与工具：命运纪事、搜索筛选、自由转盘、设置和存档管理。

## 4. PC 端方案（>= 1180px）

采用“主舞台 + 信息侧栏”的两区结构，主舞台内部再分为转盘和剧情反馈。相比当前平均三栏，主要内容更集中，角色与纪事仍可并行查看。

```text
+-----------------------------------------------------------------------+
| 品牌 / 运行状态                [返回上一转盘]  存档  [更多⋮] [新命运]|
+-----------------------------------------------------------------------+
| 姓名  路线 | 年龄/性别 | 等级/修为 | 阵营 | 核心状态 | 当前阶段      |
+---------------------------------------------+-------------------------+
| 当前任务 [修改]                             | 角色详情                |
| +----------------+  本次命运                | 武魂 / 血脉             |
| |                |  结果正文                | 天赋 / 领域             |
| |  360-420 转盘  |  概率与次数              | 魂环 / 魂骨 / 神考      |
| |                |  最近经历 1-3 条         +-------------------------+
| +----------------+                           | 命运纪事                |
| [继续剧情] [自动推进] [极速结算]            | 搜索 / 重大筛选         |
|                                             | 可滚动时间线            |
+---------------------------------------------+-------------------------+
| 高级工具与设置（默认折叠）                                            |
+-----------------------------------------------------------------------+
```

布局规则：

- 页面最大宽度保持 1580px，主舞台与侧栏建议比例为 `minmax(680px, 1.5fr) minmax(360px, .8fr)`。
- 转盘使用 `clamp(360px, 30vw, 420px)`，不再随主栏无限放大。
- 主舞台内在 1280px 以上采用左右布局，转盘在左，结果和最近经历在右。
- 右侧栏将角色详情置顶，纪事占据剩余可视高度并独立滚动。
- 关键指标摘要条横跨内容区，复用角色数据，不引入第二份业务状态。

## 5. 平板与窄桌面方案（761-1179px）

- 页面采用两栏：左侧主舞台，右侧角色详情。
- 结果面板置于转盘下方，纪事作为横跨全宽的第二行。
- 转盘限制为 340-380px。
- 顶栏允许操作按钮换行，但所有按钮保持至少 40px 点击高度。
- 900px 以下将低频的保存、读取操作收进“更多”菜单。

## 6. 移动端方案（320-760px）

移动端首屏围绕一次推进闭环设计，完整角色信息与纪事通过标签切换，不在主页面连续堆叠。

```text
+-------------------------------+
| [返回] 命运轮盘    待命    [⋮] |
+-------------------------------+
| 姓名 · 路线        45级       |
| 16岁 / 女 | 阵营 | 存活       |
+-------------------------------+
| 当前阶段                [修改]|
| 当前任务标题                  |
| 候选 12                       |
|                               |
|        240-300 转盘           |
|                               |
| 本次命运                      |
| 结果正文与概率                |
+-------------------------------+
| [主舞台] [角色] [纪事]        |
+-------------------------------+
| [继续]    [自动]    [极速]    |  <- 固定底栏
+-------------------------------+
```

交互规则：

- 默认打开“主舞台”，保留摘要、任务、小转盘和本次结果。
- “角色”展示完整 `CharacterPanel`；“纪事”展示完整 `ChroniclePanel`，切换标签不改变游戏状态。
- 标签栏使用 `role="tablist"`、方向键/点击可切换，并记住本次会话最后选择的标签。
- “返回上一转盘”固定在移动端顶栏，不收进更多菜单；无历史或正在转动时禁用。
- 底部操作栏考虑 `env(safe-area-inset-bottom)`，按钮高度至少 48px。
- 点击“自动推进”后，按钮切换为“暂停”；运行状态必须在顶栏和按钮上同时可见。
- 更多菜单固定提供“导出当前修改”；没有已应用修改时禁用并显示“暂无修改”。菜单同时承载保存、读取、新命运、导入导出和设置。
- 主内容底部留出固定栏高度，避免结果或纪事被遮挡。
- 横屏移动端按窄桌面处理，优先保证转盘与结果并排，不放大转盘。

## 7. 视觉与组件规范

### 7.1 尺寸与密度

- 断点：`mobile <= 760px`、`tablet 761-1179px`、`desktop 1180-1439px`、`wide >= 1440px`。
- 卡片圆角维持当前 6px；不新增大圆角装饰卡片。
- 正文最小字号 12px，关键结果 14-16px，移动端输入与按钮文字不小于 14px。
- 移动端可点击区域不小于 44x44px，核心推进按钮高度不小于 48px。
- 长任务名、长结果和长标签必须换行或省略，并提供可查看完整内容的方式。

### 7.2 转盘

- 保留 Canvas 绘制与现有确定性抽取逻辑，补齐按权重绘制的盘面。
- 每个扇区角度按 `optionWeight / 当前可抽选项权重总和 * 360deg` 计算；未设置 `weight` 的旧事件继续按 `1` 处理。
- 指针落点使用所选事件的累计权重区间中心，不再使用等分扇区的 `selectedIndex * segment` 公式。
- 盘面候选项、编辑器“当前占比”和抽取引擎必须复用同一份条件过滤结果，避免展示概率与实际概率不同。
- 扇区超过 16 个时减少盘内文字密度；超过 28 个继续隐藏盘内文字，以结果区作为准确文本来源。
- 极小扇区可隐藏盘内文字，但不能扩大其视觉角度；完整名称、权重和占比可在编辑器查看。
- 中心按钮随转盘缩放：PC 88-96px，移动端 72-80px。
- 转动时禁止重复触发，并保留 `aria-label` 和清晰的忙碌状态。
- 支持 `prefers-reduced-motion`：将动画压缩至 100ms 内，但结果和状态变化保持一致。

### 7.3 信息展示

- 角色摘要条新增字段：姓名、路线、年龄/性别、等级/修为、阵营/区域、存活/神考状态。
- “本次命运”保留结果色调语义，同时显示来源池、步数和精确概率。
- “最近经历”只显示最新 1-3 条，不替代完整纪事，不增加新的日志来源。
- 标签颜色继续区分武魂/血脉、天赋/领域、魂环/魂骨，但不能只依靠颜色表达类别。

### 7.4 返回上一转盘

- “一步”定义为一次已经完成结算的转盘投掷；返回后恢复到这次投掷开始前。
- 回退必须原子恢复 `MachineState`：角色属性、阶段、任务队列、活动任务、日志、上次结果、步数和 RNG 状态必须来自同一个快照。
- UI 同步恢复：根据回退后的活动任务重新加载候选项，清除选中扇区，并将转盘动画重置为待转状态。
- 转盘配置未变化时，返回后再次投掷必须得到与被撤销投掷相同的结果和概率，验证种子确定性没有被 UI 操作破坏。
- 连续返回沿用现有最多 50 份历史快照；新开命运、导入存档后清空当前撤销历史。
- 自动或极速推进中点击返回时先停止自动推进；单次转盘动画尚未结算时按钮禁用，避免恢复到半完成状态。
- 回退不新增“前进/重做”能力；用户返回后执行新的操作即形成新的时间线。

### 7.5 当前转盘编辑器

入口位于“当前阶段/当前任务”标题右侧的“修改”按钮。PC 使用居中弹窗，建议宽度 `min(920px, calc(100vw - 32px))`、最大高度 `85vh`；移动端使用全屏弹窗页，底部固定“取消”和“应用修改”。

编辑内容：

- 弹窗标题显示当前转盘名称、转盘 ID、事件数量和有效权重总和。
- 每个事件一行，包含启用开关、事件文本、权重输入、基础占比、当前占比和移除按钮。
- “基础占比”按全部已启用事件计算；“当前占比”按角色条件过滤后的实际候选事件计算，不符合条件的事件标记为“当前不可抽”。
- 权重接受有限正数，建议输入范围 `0.01-1000000`；暂时不参与抽取的事件使用启用开关，不用权重 `0` 表示禁用。
- “增加事件”在列表末尾新增空白行，本地生成稳定 ID；事件文本必填，默认启用、默认权重为 `1`。
- 新事件沿用当前 `RollTask.handler`。弹窗需要提示：等级变化、死亡、条件限制、路线分支等效果依赖现有文本解析规则，不符合规则的文本只会作为普通事件记录。
- 允许移除本地新增事件；对原始事件采用“停用”而非物理删除，保证导出时开发者能看出原始 ID 和修改意图。
- 显示脏状态；关闭、切换转盘或取消时若有未应用修改，需要二次确认。

应用规则：

- 只有当前存在可执行任务、活动池可解析且未在转动动画中时可以打开编辑器。
- 自动或极速推进中点击“修改”先暂停，再打开编辑器。
- “应用修改”前校验事件文本、权重、重复 ID，以及至少存在一个启用且当前可抽的正权重事件。
- 应用后写入独立覆盖层，以原始 `pool.id` 为主键；不直接修改被导入的 `wheels.json` 对象。
- 修改立即重建当前盘面，并作用于本局及后续新命运中再次使用的同一转盘。
- 覆盖层保存到独立的 `localStorage` 键 `douluo-wheel-overrides-v1`，不写入角色存档，不跟随“返回上一转盘”回退。
- 编辑器提供“恢复此转盘默认值”，只清除当前池覆盖；高级设置提供“清除全部转盘修改”。两者均需确认。
- 同一 RNG 状态只有在转盘配置版本不变时保证重投结果一致；返回后若主动修改了事件或权重，下一次结果允许改变，这是编辑功能的预期行为。

### 7.6 导出当前修改

- 入口固定在右上角 `[⋮]` 菜单，名称为“导出当前修改”。
- 导出范围为浏览器内所有已应用且尚未恢复默认的转盘修改，不只导出当前正在显示的一个转盘。
- 导出文件只包含被修改的转盘，但每个转盘包含最终完整事件列表，避免发送完整 273 个转盘的大文件。
- 文件名使用 `douluo-wheel-overrides-YYYYMMDD-HHmmss.json`，编码为 UTF-8。
- JSON 顶层包含格式版本、源数据版本、源导出时间、生成时间、修改池数量和 `pools` 数组。
- 每个修改池包含 `poolId`、`poolName`、原始数据指纹、最终 `options`，以及便于审阅的新增/修改/停用摘要。
- 事件至少导出 `id`、`name`、`enabled`、`weight`；本地新增 ID 使用 `local-<uuid>`，由开发者合并时决定是否换成正式 ID。
- 导出操作不清除本地修改；文件生成成功后显示文件名和修改池数量。

建议导出结构：

```json
{
  "format": "douluo-wheel-overrides",
  "formatVersion": 1,
  "source": {
    "wheelDataVersion": 1,
    "wheelDataExportDate": "2026-07-08T18:47:13.047641Z"
  },
  "generatedAt": "2026-07-15T12:00:00.000Z",
  "poolCount": 1,
  "pools": [
    {
      "poolId": "原始转盘ID",
      "poolName": "当前转盘名称",
      "baseFingerprint": "sha256-or-stable-hash",
      "summary": { "added": 1, "updated": 2, "disabled": 1 },
      "options": [
        { "id": "原始事件ID", "name": "事件文本", "enabled": true, "weight": 50 },
        { "id": "local-uuid", "name": "新增事件", "enabled": true, "weight": 10 }
      ]
    }
  ]
}
```

## 9. 实施阶段

### 阶段 A：布局骨架与基础响应式（0.5-1 天）

- 重排 `App.vue`，完成角色摘要、主舞台和侧栏结构。
- 建立四档断点与转盘尺寸约束。
- 打通返回上一转盘后的状态与盘面同步。
- 保证现有开始、转动、自动、极速、返回上一转盘和存读档功能仍可用。

交付点：PC、平板、移动端均无横向溢出，主流程可完整操作。

### 阶段 B：权重盘面与数据覆盖层（1-1.5 天）

- 统一抽取引擎、盘面和概率显示使用的候选项与归一化权重。
- 按权重绘制扇区、标签和指针落点。
- 建立按 `pool.id` 合并的本地覆盖层和持久化机制。

交付点：所有转盘的视觉扇区与实际抽取比例一致；刷新页面后已应用修改仍然生效。

### 阶段 C：当前转盘编辑与导出（1-1.5 天）

- 实现双端编辑弹窗、事件增改、启用状态、实时占比和输入校验。
- 实现单池恢复默认、全部清除和修改摘要。
- 实现右上角“导出当前修改”及稳定 JSON 格式。

交付点：修改当前池后盘面立即更新并影响抽取；导出文件可准确定位原始池并完整表达最终事件列表。

### 阶段 D：移动端专属交互（1 天）

- 实现主舞台/角色/纪事标签页。
- 实现顶栏更多菜单和底部固定操作栏。
- 处理安全区、虚拟键盘、长内容滚动和横屏。

交付点：375x667 视口内可看到角色摘要、当前任务、完整转盘和本次结果标题；无需滚动即可触达三个推进操作。

### 阶段 E：信息增强与状态反馈（0.5 天）

- 增加最近经历摘要和更清晰的运行状态。
- 优化结果色调、空状态、结束状态和自动推进中的按钮反馈。
- 补齐长文本、极端候选数和无数据状态。

交付点：用户在任意阶段都能判断“当前是谁、正在做什么、刚发生什么、下一步能做什么”。

### 阶段 F：验证与收尾（0.5-1 天）

- 运行类型检查、单元测试和生产构建。
- 进行桌面与移动端截图回归和核心交互冒烟测试。
- 检查键盘导航、焦点、对比度、减弱动画和屏幕阅读器标签。

交付点：满足第 10 节验收标准，无 P0/P1 展示或交互缺陷。

预估总工期：4.5-6.5 个开发日，不含新增美术资产和剧情功能。

## 10. 验收标准

### 功能回归

- 三条起始路线均可创建，手动推进、自动推进、极速结算与暂停可用。
- 返回上一转盘、本地保存/读取、导入/导出存档和导出传记可用。
- 返回后任务标题、候选项、角色属性、结果、日志和步数同时恢复，不出现前后步骤信息混合。
- 转盘配置未变化时，对同一步执行“投掷 -> 返回 -> 再投掷”，两次结果与概率完全相同。
- 连续完成 3 次投掷后可逐步返回 3 次；返回到历史起点后按钮正确禁用。
- 自动推进时不能产生并发回退；动画结算期间返回按钮保持禁用。
- 页面重排不改变相同种子和操作顺序下的抽取结果。
- 结束横幅不遮挡移动端底栏，重新开始入口可操作。

### 权重编辑与导出

- 权重为 `1:3:6` 的三个无条件事件，盘面角度与实际概率分别为 `10%:30%:60%`，允许合理的像素取整误差。
- 修改权重并应用后，当前盘面无需刷新立即更新，后续抽取使用新权重。
- 新增事件后获得稳定本地 ID，可以参与当前转盘抽取；刷新页面后仍存在。
- 原始事件可以停用，本地新增事件可以移除；恢复默认后原始数据与盘面完全恢复。
- 编辑器拒绝空事件文本、非有限权重、非正权重以及没有任何当前可抽事件的配置。
- 返回上一转盘后能编辑恢复出的当前池；配置未改变时重投结果一致，配置改变后按新权重抽取。
- 导出 JSON 通过结构校验，包含正确源版本、池 ID、池名、修改摘要和最终事件列表。
- 两个转盘分别修改后，单次导出同时包含两个池；恢复其中一个默认值后再次导出只包含剩余池。

### 响应式

- 验证视口：320x568、375x667、390x844、768x1024、1024x768、1366x768、1440x900。
- 所有视口无页面级横向滚动，无按钮、文字、转盘和浮层互相遮挡。
- 375x667 下转盘直径不超过 280px；1440x900 下不超过 420px。
- 移动端底栏不遮挡最后一条内容，iOS 安全区有正确留白。
- 从移动端切换到桌面端后，面板顺序和可见内容正确，不丢失 UI 状态或游戏状态。

### 可用性与可访问性

- 核心操作点击区域至少 44x44px，移动端主要按钮高度至少 48px。
- 键盘可完成标签切换、菜单开关、打开编辑器、修改权重、增加事件、开始新命运和推进剧情。
- 菜单与弹窗打开后焦点进入，关闭后焦点返回触发元素。
- 自动推进、忙碌、暂停和终局状态有文本表达，不仅依赖颜色或动画。
- 启用 `prefers-reduced-motion` 后没有长时间旋转动画。

### 工程质量

- `npm test` 全部通过。
- `npm run build` 全部通过。
- 自动化测试覆盖连续返回、返回后重投一致、动画期间禁用和自动推进停止。
- 自动化测试覆盖权重归一化、加权扇区落点、覆盖合并、持久化恢复、编辑校验和导出结构。
- 不复制游戏状态，不新增与状态机并行的业务状态源。
- 关键响应式行为至少有一组组件测试或端到端冒烟脚本覆盖。

## 11. v0.1 不包含

- 不主动改写 273 个内置转盘及 2,456 个原始选项的默认内容和权重；用户修改只进入本地覆盖层。
- 不新增登录、云存档、排行榜、分享海报和多人功能。
- 不进行完整品牌换肤，不引入插画、角色立绘或新的视觉资产管线。
- 不重构有限状态机，不更改现有存档版本。
- 不在 v0.1 增加“前进/重做”时间线。
- 不允许浏览器直接改写仓库中的 `src/data/wheels.json`；修改通过覆盖层运行并以 JSON 交给开发者合并。
- 不在 v0.1 提供修改 JSON 的导入、在线提交或自动合并能力。
- 不在本轮实现 PWA 安装、离线缓存和推送通知。

## 12. 建议开发顺序

1. 先统一实际候选项与权重计算，完成按比例绘制和正确落点。
2. 再建立本地覆盖层，保证查询、抽取、盘面和持久化读取同一份有效数据。
3. 实现当前转盘编辑器、恢复默认和修改 JSON 导出。
4. 完成双端布局、移动标签、更多菜单和返回上一转盘交互。
5. 补充最近经历、状态反馈、极端内容和配置变更后的回退规则。
6. 最后统一做自动化测试、多视口截图回归、功能冒烟与可访问性检查。

此顺序能让每个阶段都保留可运行版本，也便于在视觉验收后再进入下一阶段。
