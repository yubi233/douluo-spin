# v0.3 原版转盘恢复与长程验证标准

## 状态与适用范围

本标准适用于从 `1338d1b^:src/data/wheels.json` 恢复的全部原版内容。它是当前 `v0.3` 的发布前置条件，优先于任何历史 400 份认证证据；历史版本的长程通过记录不得用于证明恢复后的内容已生效。

恢复完成不是“生成文件中仍有 276 个池”，而是每个原始池及其选项均被保留、具有明确的 v0.3 ID 映射，并由实际流程在正确条件下调度。用单项占位池替代原始池，或只在内容浏览器中可见但永远不能进入游戏流程，均视为恢复失败。

### 魂兽成神边界

- 纯魂兽路线（`route: beast`）不进入人类神位感召、神考、海神岛继承或人类自创神位流程。
- 纯魂兽只有在百万年雷劫成功后，获得 `entity.godhood.beast` 所代表的自创兽神神位，才可产生存活成神终局；此前或其他神位授予、存活成神终局均为无效事件。
- 海神时代魂兽的原版时间线终局仍以该百万年成功为前置，不构成海神继承路径。
- 百万年池中“突破成功但未获神位”的原版选项只能进入“百万年长生”非成神终局，不得授予 `godhood`，也不得计入任何成神统计。

## 不可变原始基线

- 来源：`src/content/v03/legacyContent.source.json`。
- 来源校验：SHA-256 必须为 `a15531384c18d428622596d1d96b645ded8a8d65e14e3d948c1c33ad1c024b1b`。
- 基线规模：276 个池、2,500 个选项、25 个标签。
- 每个原始池和选项必须保留可追溯的一对一 ID 映射：`sourcePoolId -> activePoolId`、`sourceOptionId -> activeOptionId`。扩展内容只能以独立扩展记录加入，不能替换、合并或静默删除原始记录。
- 每个映射必须比较原始 `name`、`enabled` 与基础权重；题名和权重不得被占位值覆盖。唯一允许的权重差异必须来自归档 `catalog.ts` 的明确修正规则，并在账本中同时记录源权重、活动权重、规则来源和理由；当前仅有“唐三6岁”从 350 修正为 20。结构化效果和条件可以在 v0.3 中新增，但不能改变该原始盘面本身的候选集合。

## 三层恢复门禁

### R1：静态逐项保真

在 `npm run content:migrate` 和 `npm run content:validate` 中生成并校验 `pool-recovery-ledger.json`。该清单至少包含原始池 ID、原始题名、活动池 ID、原始/活动选项数、所有选项映射、扩展项、迁移规则状态和来源校验。

R1 通过条件：276/276 原始池、2,500/2,500 原始选项和 25/25 标签均有映射；任何计数差异、重复映射、缺失项、替代项或未说明扩展均失败。`器武魂`、`兽武魂`入口必须保留完整原始候选集，不能以“青钢剑武魂池”一项代替。

### R2：真实流程可达性

所有 276 个原始池必须在 `pool-recovery-ledger.json` 中具有一个且仅一个流程状态：

| 状态 | 含义 | 是否允许关闭恢复门禁 |
|---|---|---|
| `observed` | 在真实用户流程的确定性运行中被调度并抽取 | 是 |
| `controlled` | 未落入 100 次随机样本，但由真实 Process/条件链、固定 seed 或可审计的前置状态进入并抽取 | 是 |
| `composed` | 原始父池以旧版分类协议拆为分类转盘和子转盘；子池选项与权重的并集逐项等于原始父池，真实流程通过该链完成抽取 | 是，仅限 `兽武魂`、`器武魂` |
| `blocked` | 仍缺少流程调度、条件、前置数据或效果语义 | 否 |

“controlled” 不能通过直接向 agenda 注入一个池、直接调用 draw 或编辑状态伪造。证据必须包含从 `run.start` 开始的 EventBatch、每一回合的 `poolId`/`optionId`、满足的前置条件和最终状态。每个池还要记录所属入口、可能路线、人类/魂兽/化形条件、触发流程和一个可复现 seed。

设置类入口必须额外按类型矩阵验证：本体、变异、概念型、极致直接调度完整原始池；兽、器必须严格执行 `武魂类型 -> 武魂分类转盘 -> 分类子转盘`。两类分类子池的选项题名和基础权重并集必须逐项等于对应原始父池；选中任何一个原始武魂后必须发出 `signal.setup.martial-soul-selected` 并继续进入年龄设定。此项专门防止“流程可跑通，但被调度到单项占位池”的回归。

### R3：100 次长程频率与覆盖报告

恢复期间运行两组固定的 100 次确定性长程样本，而不是直接启动正式的 20 x 20 发布认证：100 human 用于人类成长、成神率和剧情验证，100 beast 用于魂兽与化形验证。化形结果单独计数。每个样本保存传记、完整 JSON 归档、EventBatch、trace 和自动审计结果。

每次运行都必须将所有实际抽取写入 `pool-frequency.json`，同时输出包含全部 276 个原始池的完整频率表。每一行至少包含：

```json
{
  "sourcePoolId": "<原始池 ID>",
  "activePoolId": "<v0.3 池 ID>",
  "title": "<原始题名>",
  "runsHit": 0,
  "drawCount": 0,
  "runFrequency": 0,
  "drawFrequency": 0,
  "sampleSeeds": [],
  "coverageStatus": "observed | controlled | blocked"
}
```

定义如下：

- `runsHit`：同一条路线的 100 个样本中至少一次抽到该池的样本数。
- `drawCount`：该池实际被抽取的总次数；循环池可以大于 `runsHit`。
- `runFrequency`：`runsHit / 100`。
- `drawFrequency`：`drawCount / 所有原始池的总抽取次数`；若分母为零则报告失败，不能填充为 0。
- `sampleSeeds`：最多保留 10 个命中 seed，完整命中清单放入按池索引的证据文件。

零频条目必须保留，且只能以 `controlled` 或 `blocked` 说明；不得从报告中删除。R3 的通过条件是 human 100/100、beast 100/100 自动审计通过，频率表恰好覆盖 276 个源池，所有零频池都有真实受控证据，`blocked = 0`。频率不是权重正确性的统计证明，权重保真由 R1 的逐项比较保证。

## 每样本数值与副作用审计

每一个长程样本必须执行以下不可跳过的自动审计，并将逐回合结果写入归档的 `side-effect-audit` 段。仅比较最终 state 不足以证明中途事件生效。

### 等级与战力

- 每个 EventBatch 必须从前一状态可重放到后一状态；每一条 `stat.changed` 的 `before`、`after` 和 clamp 结果必须与该回合被抽中选项的结构化 effect，或明确列入白名单的 Process 事件完全一致。
- 人类非神考状态的等级必须位于 `1..max-level`；进入或完成神考后，等级必须位于 `100..159` 且具有对应的 god-trial/godhood/终局事件。所有路线的绝对边界为 `1..159`。不允许由文案、隐式修正或未记录的 Process 改写等级。
- 魂环、领域、魂骨、武魂品阶和天赋/战斗特质改变时，必须同步重新计算并记录战力。当前只以 `level * level / 20` 作为 `progression.combat-power` 的实现不满足恢复门槛；须将原版 `calculateCombatPower` 的组成恢复为结构化、纯函数的 v0.3 规则：等级基数、魂环、武魂品阶、领域、魂骨及天赋/战斗特质系数。运行时不得重新解析中文文案。
- 每次涉及战斗、剧情战斗、等级、魂环、领域、魂骨、武魂或战斗特质的事件，都必须记录 `combatPowerBefore`、`combatPowerAfter`、各组成项和触发项。审计以同一纯函数重算，数值必须完全相等；存活的人类魂师战力不得为 0 或非有限数。
- 一旦恢复了结构化的对手等级/战斗结果，胜负事件还必须验证双方战力关系和明确的越级规则。尚未结构化的对手或战斗条件必须标为 `blocked`，不能被“剧情文本存在”视为通过。

### 成神率校准

- 只以 100 个 human 样本作为分母；魂兽、化形前阶段和人工控制场景不参与计算。
- `ending.god-ascension` 与 `ending.self-created-ascension` 均计为成神；早死、普通终局、未完成和缺失终局均不计为成神。
- `god-rate.json` 必须记录完整 seed 集、成神/非成神 ending、成神路径、分子、分母和比例。通过区间固定为 `45% <= godRate <= 55%`，即 100 个样本中 45 至 55 个成神。
- 该比例由恢复后的结构化权重、资格和流程共同决定。任何为满足比例而在测试中改 seed、跳过失败样本、强制结局或在运行后筛选样本的行为都使整组结果无效。

### 每个事件的副作用与传记抽检

- 对每个被抽中的 `poolId/optionId`，审计器在抽取前快照上调用同一结构化 effect 编译逻辑，并把期望 effect 事件与实际 EventBatch 中由该选项产生的事件逐项比较。必须覆盖 `stat.change`、`entity.grant`、`entity.revoke`、`time.advance`、`signal.emit` 和 `run.finish`；Process 后续调度事件与选项副作用分开记录。
- 逐项比较后必须重放整个 EventBatch，并验证每个副作用都落入下一状态：属性值、实体集合、时间线、信号引发的任务、终局与传记投影一致。空 effect 只能对应显式纯叙事选项；原版文案声称的副作用尚未结构化时必须是 `blocked`，不能以 `effects: []` 通过。
- 人工传记抽检使用结果生成后的固定选择器，至少审阅每组 100 样本的 20 份（20%），并覆盖全部已出现的 effect 类型、至少一条成神、一条死亡、一条化形以及最高/最低战力样本。每份抽检传记须逐事件对照 JSON 的 EventBatch 和 `side-effect-audit`，确认传记没有把未落地副作用写成事实，也没有遗漏已落地的关键副作用。
- 任一自动副作用不一致、等级/战力审计失败、传记与事件不一致或未满足抽检覆盖，均使相应恢复代次失败。

## 强制产物与门禁

新入口命令在实现后命名为 `npm run simulate:recovery`。它每次只能生成一个包含双 100 cohort 的固定恢复代次，写入：

```text
simulation-archives/v03-recovery/<generation>/
  manifest.json
  runs/
  pool-frequency.json
  pool-recovery-ledger.json
  pool-coverage.json
  findings/
  audit-report.md
```

`manifest.json` 必须记录源 SHA-256、commit、contentVersion、200 个 seed、路线分布、静态保真结果、所有池的覆盖状态汇总、频率报告 SHA-256、等级/战力审计汇总、成神率和传记抽检索引。任何内容、流程或规则修复均会使该代次失效；必须使用新的 `contentVersion` 从新的两组 100 次样本重新开始。

启动恢复代次前，工作树必须干净：不得有已跟踪修改或未跟踪文件。`npm run simulate:recovery` 必须在创建目录或写入归档前校验该条件；未满足时直接失败，不能把工作树代码标记为某个 `HEAD` commit 的认证证据。

`npm run test:verify` 只能证明当前自动化门禁通过。只有 R1、R2、R3 同时通过且迁移 backlog 为零，才允许进入正式 20 x 20 认证。当前 backlog 不为零时，验证结果必须显示 `recovery-in-progress`，不得称为“长程恢复通过”。

## 缺陷处置

以下任一情况均为 P0 恢复缺陷：原始池或选项缺失、活动流程调度占位池、池无法由真实流程到达、频率报告漏池、无归因零频、空候选、效果未推动预期流程、或 100 次样本中任一自动审计失败。

发现后必须记录最小 seed、pool/option ID、触发回合、来源规则、根因和回归测试 ID；修复后重新执行 R1、R2、R3，不得把修复前频率或覆盖记录混入新代次。
