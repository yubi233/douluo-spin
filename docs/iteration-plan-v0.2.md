# 迭代计划 v0.2：剧情线数值化与武魂分级体系

> 状态：起草中
> 版本：v0.2
> 日期：2026-07-15
> 依赖诊断文档：`docs/storyline-analysis.md`、`docs/unmodified-character-bugs.md`、`docs/unrewarded-storylines.md`、`docs/武魂分类清单.md`、`docs/combat-power-system.md`

---

## 1. 迭代目标

v0.2 聚焦于将纯文本剧情的战斗结果、势力经历、学院选择和魂兽遭遇**数值化落地**——让每一条剧情选项都能实际修改角色状态（等级、容貌、修为、称号、势力等），并基于武魂分级体系影响先天魂力、战力和成神概率。

核心目标：

1. 补全 handler 链路：修复 `beastEncounter` 缺失，确保魂兽遭遇的修为变化生效。
2. 补全全局反馈：`applyCommon()` 增加容貌处理，使神考奖励的容貌提升不再落空。
3. 剧情数值化：为主线战斗结果（嘉陵关/神战/大赛）增加等级变化和称号奖励，让"无伤战胜"和"重伤战死"产生区别。
4. 势力反馈化：为势力专属剧情和学院经历嵌入可解析的等级/称号变化。
5. 武魂分级落地：六阶武魂分级体系影响先天魂力分布（正态加权）、战力值和神位条件。
6. 战力系统部署：独立武魂战力、魂环分段战力、魂骨年限战力，并以系数方式纳入天赋和称号的影响。

## 3. 实施阶段

### 阶段 A：Handler 链路补全（BUG-01, BUG-02）

**目标**：修复程序层面的反馈缺失，让剧情结果能正确修改角色数据。

**范围**：仅 `src/domain/machine.ts`

**任务**：
- A.1 新增 `case 'beastEncounter':` handler，解析修为增减（`修为+N年`、`修为掉落N年`、`修为-N万年`）并写入 `context.beast.cultivation`
- A.2 在 `applyCommon()` 中新增 `容貌+/-数字` 的解析和容貌等级升降（F→E→D→C→B→A→S→EX，检查 EX 上限）

**验收**：魂兽遭遇抽中 `"你遇到万年魂兽，修为+1000年"` 后 `context.beast.cultivation` 增加 1000；神考奖励抽中 `"容貌+1"` 后 `context.appearance` 从 B 升至 A。

**估时**：0.5 天

---

### 阶段 B：剧情数值化——战斗结果（BUG-04）

**目标**：让主线剧情中的战斗选项产生有差别的数值变化。

**范围**：`src/data/wheels.json`（嘉陵关、神战、大赛、遭遇剧情池）

**任务**：
- B.1 为战斗结果选项嵌入可被 `applyCommon()` 解析的文本模式：
  - 无伤战胜 / 轻松战胜 / 秒杀 / 斩杀 → `等级+1` 或 `等级+2`
  - 轻伤 → 不变
  - 重伤 / 战败（非致命）→ `等级-1`
  - 战死 保留现有 `lethal()` 检测
  - 特殊成就（1打7、单刷等）→ `获得【XXX】称号`
- B.2 检查现有 `lethal()` 正则在战斗池中的覆盖率（"被XX重创""被围攻至死"是否被捕获）
- B.3 嘉陵关12个子池统一规则，神战8池统一规则

**验收**：同一场嘉陵关战斗，抽到"无伤斩杀军团"获得等级+1，抽到"轻伤无功无过"无事发生，抽到"重伤被围攻至死"触发死亡结局。

**估时**：1.5 天

### 阶段 C：剧情数值化——势力与学院（BUG-05, BUG-06）

**目标**：让势力专属剧情和学院经历产生称号/等级变化。

**范围**：`src/domain/canonAdditions.ts`（势力硬编码） + `src/data/wheels.json`（学院池）

**任务**：
- C.1 修改8个势力的120条选项文本，嵌入可解析模式：

  - 被孤立 → `获得【孤独者】称号`

- C.3 确保"加入武魂殿招募"的 `叛出史莱克` 逻辑在势力字段中体现

**验收**：在势力剧情中抽到"接任团长，重整章程"后，角色等级+2；在校2年选择"拜院长为师"后，角色获得 `【院长亲传】` 称号。

### 阶段 H：终极长程回归测试

**目标**：运行 100 次完整全流程（人类 50 + 魂兽 50），逐份分析每份人物传记，使用增强审计系统检测所有已知 BUG 类别。

**范围**：`scripts/run-long-simulation.ts`（扩展）、`src/domain/simulation.ts`（增强审计）

**任务**：
- H.1 将采样数从 20 提升到 100（50人类 + 50兽），启用 `samplingMode: 'weighted'`（与玩家实际体验一致）
- H.2 增加传记级审计规则（逐份分析人物传记文本）：
  - 空池报错检测（任何 `throw Error` → 标记）
  - 武魂阶位缺失（传记中武魂无阶位标记）
  - 容貌未变化（从创角到终局 `appearance` 值不变）
  - 等级/修为越界（level ≤ 0 或 > maxLevel，cultivation 为负）
  - 战力值异常（为 0 或过于异常的值）
  - 神考条件不匹配（武魂阶位与神位条件冲突）
  - 内部 flag 泄露到传记文本中（`_badCount`、`_god99Triggered` 等）
  - 势力历史不完整（`factionHistory` 为空或格式异常）
  - 称号/领域/魂骨计数为 0 但传记中标注了对应节
- H.3 增加 trace 级审计规则：
  - beastEncounter 触发后 `cultivation` 未变化
  - growth 中 `容貌+1` 触发后 `appearance` 未变化
  - 战斗剧情（嘉陵关/神战）`无伤战胜` 后等级未变化
  - 势力剧情 `被表扬` 后 trait 未变化
  - 连续 3 次 time jump 后 level 无变化（可能卡死）
- H.4 输出详细报告：`simulation-archives/<timestamp>/audit-report.md`，包含：
  - 总通过率
  - 按 BUG 类别分组的失败统计
  - 每份失败的传记摘要 + 关键问题
  - 所有成功传记的统计分布（等级分布、武魂分布、终局类型等）

**验收**：100 份传记全部通过增强审计（0 个致命 BUG），审计报告可追溯每项失败的具体传记和问题行。

**估时**：1.5 天

## 4. 验收标准

### 功能验收

- beastEncounter 触发后，`context.beast.cultivation` 按修为+N年/修为掉落N年变化
- 容貌+1 触发后，`context.appearance` 按 F→E→D→C→B→A→S→EX 升级
- 主线战斗结果的"无伤战胜""轻伤""重伤战死"选项分别有等级+1/不变/等级-1的效果
- 势力剧情抽到"被团长表扬"获得对应称号，抽到"接任团长"等级+2
- 学院经历抽到"拜院长为师"获得称号，"默默闭关"等级+3
- 武魂阶位在角色面板可见
- 战力值在角色面板可见，随等级/魂环/武魂变化实时更新
- 草鞋+90级触发专属剧情获得双枪武魂

### 工程质量

- `npm test` 全部通过
- `npm run build` 全部通过
- 数据量：武魂总数 ≥ 259，总选项数 ≥ 2,488
- 战力公式为纯函数，相同输入 → 相同输出
- 不破坏现有种子确定性

### 数据完整性

- 新增武魂不在内置转盘中创建重复 ID
- 新增剧情选项不破坏现有分支结构
- 称号/领域/魂骨的 `addUnique` 不产生重复

## 5. v0.2 不包含

- 不新增斗罗2/3/4时代的武魂、魂灵、斗铠、魂导器
- 不修改有限状态机的迁移结构
- 不改变存档 JSON 格式
- 不新增 UI 大改（战力显示除外）
- 不增加神位体系的数量（保持四级神位不变）
- 不新增联机、云存档、排行榜功能

## 6. 建议开发顺序

1. **阶段 A** — Handler 补全（基础链路，影响最大）
2. **阶段 B** — 战斗数值化（嘉陵关/神战，选项最多）
3. **阶段 C** — 势力与学院数值化（硬编码文本改）
4. **阶段 F** — 补充武魂（可并行）
5. **阶段 D** — 武魂分级 UI 显示
6. **阶段 E** — 战力 UI 显示
7. **阶段 G** — 单元测试与构建
8. **阶段 H** — 终极长程回归测试（100次全流程）

此顺序保证每一阶段都有可运行的完整游戏版本，且在 A 完成后再做 B/C 可以验证 applyCommon 的新能力是否对嵌入的文本生效。
