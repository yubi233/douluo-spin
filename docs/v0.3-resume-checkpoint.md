## 5. 恢复顺序

恢复后先确认工作树没有被其他操作改写：

```bash
git status --short
git rev-parse HEAD
```

然后从被中断的浏览器门禁继续：

```bash
npm run test:e2e
```

E2E 通过后，依次处理编辑器结构化控件、`ContentService` 测试、生产架构扫描、`test:verify` 内容校验接入，再运行：

```bash
npm run content:validate
npm run test:verify
```

完成上述回归后才能开始长模拟预演。预演缺陷修复会使旧预演代次失效；最终认证只能在候选 commit 和 `contentVersion` 冻结后开始。

## 6. 最终认证不可放宽的规则

- 每次只运行一个批次，不循环、不并行。
- 每批 20 条，固定 10 human + 10 beast。
- 必须逐份阅读传记和 JSON 归档，再写人工审阅结果。
- 前一批 20 条全部审阅并通过 gate 后，才能生成下一批。
- 任意代码或内容修复都会使当前认证 generation 失效。
- 最终有效产物必须来自同一 commit、同一 `contentVersion`，共 20 批、400 条。
- 所有审阅状态必须为 passed，不得存在 pending 或 failed。
