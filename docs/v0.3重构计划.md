> 状态：恢复实施中；阶段 A-G、2 x 20 预演和阶段 H 的 `v0.3.0` 认证已作为历史证据封存。原始池已逐项恢复至 `v0.3.8`；最新双 100 审计为 200/200 自动通过、46% human 成神率、210 observed / 39 controlled / 27 blocked。R2/R3 尚未关闭，恢复完成后必须重新认证。

## 1. 执行摘要

v0.3 将当前“有限状态机 + 选项文本解析 + 全局上下文修改”的实现，重构为一个纯 TypeScript、确定性、可回放的事件驱动游戏内核。

目标架构由五个模式组成：

1. **Statechart** 管理游戏生命周期和合法命令。
2. **Command / Domain Event** 管理一次转盘回合的决策、事实和原子提交。
3. **Specification / Expression** 管理候选资格、动态权重和领域守卫。
4. **Effect Interpreter** 将内容定义的结构化效果编译为领域事件。
5. **Process Manager** 响应领域事件，编排人类、魂兽、化形、魂环、神考、剧情和终局等跨回合流程。

v0.3 不以“保持旧代码内部行为”为验收目标，但要保留现有产品的核心用户旅程：新建命运、人类/魂兽/化形路线、加权抽取、自动推进、回退、存档、转盘编辑、人物传记和长程模拟。

实施不按“先建全部内核，再一次性接入内容”的水平分层方式推进，而是按可运行的领域纵向切片交付。每个切片必须同时包含内容 schema、规则、effect、Process Manager、投影和自动化测试，避免在重构后期才发现契约无法承载真实剧情。

最终发布认证必须由 `scripts/run-long-simulation.ts` 在同一代码提交和同一内容版本上完成 **20 批 × 每批 20 次，共 400 次加权全流程模拟**。每一次模拟的传记和归档都必须被逐份审阅，不允许仅依赖汇总统计或抽样检查。

## 2. 背景与问题边界

### 2.1 当前架构问题

| ID | 问题 | 直接影响 |
|---|---|---|
| V03-PROBLEM-01 | `WheelOption.name` 同时承担展示文案和业务指令 | 改文案可改变属性、死亡、候选资格和后续队列 |
| V03-PROBLEM-02 | `machine.ts` 同时承担状态迁移、文本解析、领域修改和流程编排 | 新增规则容易产生跨路线回归 |
| V03-PROBLEM-03 | 转盘以名称查找，并存在 `includes` 模糊回退 | 重名或近似名称可静默解析到错误池 |
| V03-PROBLEM-04 | 游戏实体和状态大量使用中文字符串作为身份 | 改名、同名、别名和本地化都会传导到业务逻辑 |
| V03-PROBLEM-05 | `flags: Record<string, string | number | boolean>` 是无约束逃生口 | 不可发现的键名冲突、状态遗漏和存档污染 |
| V03-PROBLEM-06 | 结果结算不是显式事务 | 多个效果和后续任务的失败边界难以证明 |
| V03-PROBLEM-07 | 编辑器允许改文本，但不能显式编辑条件和效果 | 用户无法知道一段文本将产生什么业务后果 |
| V03-PROBLEM-08 | 深拷贝快照承担回退和存档原子性 | 可运行，但缺少可审计回合边界和完整回放能力 |

### 2.2 本轮核心判断

- 不继续扩充文本正则。
- 不把旧正则搬进“通用文本解析器”。
- 不建立按 2,000+ 选项 ID 分支的巨型 `switch`。
- 不允许内容 JSON 通过任意属性路径或 JavaScript 修改状态。
- 允许一次性工具将旧内容转换为 v0.3 草案，但转换产物必须成为新的唯一事实来源，运行时不得保留文本兼容层。

## 5. 架构原则与边界

### 5.1 运行时边界

v0.3 仍是纯浏览器 Vue 3 应用，不新增服务端边界。

| 边界 | 责任 | 不负责 |
|---|---|---|
| Vue UI | 展示投影、收集命令、播放转盘动画 | 抽取、资格、效果、流程和 RNG |
| Application Service | 命令入口、事务编排、持久化和 UI 通知 | 内容文案解析 |
| Core | 状态图、规则、抽取、效果、事件、reducer、流程 | Vue、DOM、localStorage、下载 |
| Content | 池、选项、实体、结局、显示文案和机制声明 | 任意 JavaScript、任意状态路径 |
| Infrastructure | localStorage、JSON 文件、下载和时钟适配器 | 领域规则 |

### 5.2 单一事实来源

- 当前游戏事实的权威来源是已提交的 `DomainEvent[]`。
- `GameState` 是事件日志的投影，可随时从初始状态重放。
- Statechart 生命周期与世界投影在同一个 `GameState` 中，不建立并行的第二份游戏状态。
- Vue 只持有 `GameViewModel` 和动画/弹窗等短命 UI 状态。

### 5.3 展示与机制物理分离

内容编译器将源内容分成两个运行时索引：

```text
ContentSource
  ├── PresentationCatalog  -> UI / 传记 / 日志
  └── MechanicsCatalog     -> Core / 抽取 / 规则 / 效果
```

`MechanicsCatalog` 的 TypeScript 类型中不存在 `name`、`title`、`description` 字段。这是编译期隔离，不仅是代码约定。

## 6. 目标数据流

```text
UI dispatch(Command)
        |
        v
GameService
  1. Statechart 验证命令
  2. DrawEngine 计算候选与落点
  3. EffectCompiler 编译选项效果
  4. ProcessManagers 消费事件并生成后续事件
  5. Reducer 在隔离状态上验证整个事件批次
  6. 原子追加 EventBatch
        |
        +--> GameState projection
        +--> GameViewModel
        +--> Persistence adapter
        +--> UI animation receipt
```

转盘动画不再是领域状态。`SPIN` 命令立即产生已提交的回合收据，UI 按收据中的 optionId 和落点播放动画。页面在动画中关闭时，重新打开可直接显示已提交结果，不存在“半结算”存档。

## 7. 核心契约

### 7.1 稳定 ID

```ts
type Brand<T, Name extends string> = T & { readonly __brand: Name }

type PoolId = Brand<string, 'PoolId'>
type OptionId = Brand<string, 'OptionId'>
type EntityId = Brand<string, 'EntityId'>
type PolicyId = Brand<string, 'PolicyId'>
type SignalId = Brand<string, 'SignalId'>
type EndingId = Brand<string, 'EndingId'>
type TurnId = Brand<string, 'TurnId'>
```

- ID 使用语义化、稳定、ASCII 字符串，例如 `pool.setup.gender`、`trait.twin-souls`。
- 中文名称仅存在 `PresentationCatalog`。
- 相同字符串不能在不同 ID 类型之间隐式传递。

### 7.2 内容定义

```ts
interface OptionSource {
  id: OptionId
  presentation: {
    title: string
    description?: string
  }
  mechanics: {
    baseWeight: number
    availableWhen?: Predicate
    weightModifier?: NumberExpression
    effects: EffectSpec[]
  }
}

interface PoolSource {
  id: PoolId
  presentation: { title: string; description?: string }
  tags: EntityId[]
  options: OptionSource[]
}
```

### 7.3 条件和数值表达式

```ts
type Predicate =
  | { type: 'all'; items: Predicate[] }
  | { type: 'any'; items: Predicate[] }
  | { type: 'not'; item: Predicate }
  | { type: 'compare'; fact: FactKey; op: CompareOperator; value: Scalar }
  | { type: 'contains'; fact: CollectionFactKey; value: EntityId }
  | { type: 'policy'; policyId: PolicyId; args?: JsonObject }

type NumberExpression =
  | { type: 'constant'; value: number }
  | { type: 'fact'; fact: NumericFactKey }
  | { type: 'add' | 'multiply' | 'min' | 'max'; items: NumberExpression[] }
  | { type: 'clamp'; value: NumberExpression; min: number; max: number }
  | { type: 'policy'; policyId: PolicyId; args?: JsonObject }
```

约束：

- `FactKey` 不是自由属性路径，必须由 Fact Registry 注册。
- 通用操作符只处理无副作读操作。
- 复杂战力、神位资格等逻辑使用带 ID 的纯函数 policy，不扩展成通用脚本语言。

### 7.4 效果与领域事件

```ts
type EffectSpec =
  | { type: 'stat.change'; stat: StatId; delta: NumberExpression }
  | { type: 'entity.grant'; entityType: EntityType; entityId: EntityId }
  | { type: 'entity.revoke'; entityType: EntityType; entityId: EntityId }
  | { type: 'time.advance'; years: NumberExpression }
  | { type: 'signal.emit'; signalId: SignalId; payload?: JsonObject }
  | { type: 'run.finish'; endingId: EndingId }

type DomainEvent =
  | { type: 'option.selected'; poolId: PoolId; optionId: OptionId; probability: number }
  | { type: 'stat.changed'; stat: StatId; before: number; after: number }
  | { type: 'entity.granted'; entityType: EntityType; entityId: EntityId }
  | { type: 'entity.revoked'; entityType: EntityType; entityId: EntityId }
  | { type: 'time.advanced'; before: number; after: number }
  | { type: 'task.scheduled'; task: Task }
  | { type: 'phase.changed'; from: GamePhase; to: GamePhase }
  | { type: 'run.finished'; endingId: EndingId; alive: boolean }
```

`EffectSpec` 表示“内容希望应用什么”，`DomainEvent` 表示“经过规则、上限、免疫和守卫后实际发生了什么”。内容不能直接伪造 `before/after` 领域事件。

### 7.5 命令和回合收据

```ts
type GameCommand =
  | { type: 'run.start'; route: StartRoute; seed: string }
  | { type: 'turn.spin' }
  | { type: 'turn.undo' }
  | { type: 'run.finish'; endingId: EndingId }
  | { type: 'run.reset' }

interface EventBatch {
  turnId: TurnId
  command: GameCommand['type']
  contentVersion: string
  rngBefore: number
  rngAfter: number
  events: DomainEvent[]
}
```

- `turn.spin` 是一个原子回合。
- Effect Compiler、Process Manager 和 Reducer 先在隔离投影上运行到稳态，然后一次性追加 `EventBatch`。
- Process Manager 反应必须有最大步数；超限抛出 `ProcessCycleError` 并取消整个批次。

### 7.6 世界状态

```ts
interface GameState {
  schemaVersion: 3
  contentVersion: string
  phase: GamePhase
  actor: ActorState
  timeline: TimelineState
  progression: ProgressionState
  agenda: TaskAgenda
  random: RandomState
  turn: number
  ending: EndingState | null
}
```

领域子状态使用明确类型，不再提供全局 `flags` 字典。临时跨阶段信息放入具体流程状态，例如 `GodTrialProgress`、`StoryProgress`、`BeastProgress`。

### 7.7 错误契约

| 错误 | 触发条件 | 处理 |
|---|---|---|
| `ContentValidationError` | schema、ID、引用或注册表无效 | 构建失败；开发环境列出完整路径 |
| `InvalidCommandError` | 当前 Statechart 不接受命令 | 不修改日志；UI 恢复可操作状态 |
| `NoEligibleOptionError` | 当前池无合法候选 | 不回退到“忽略条件”；记录池、任务和 fact 摘要 |
| `UnhandledEffectError` | effect 未注册 | 取消批次，不允许静默忽略 |
| `ProcessCycleError` | 流程反应未在上限内稳定 | 取消批次并输出事件链 |
| `ContentVersionMismatch` | 存档内容版本与当前包不同 | 明确拒绝载入，不做隐式迁移 |

## 8. Statechart 与流程编排

### 8.1 外层 Statechart

计划引入 XState 5 管理外层状态图，仅使用其层级状态、guard 和类型化事件能力。领域效果、候选规则和内容修改不写入 XState action。

```text
idle
running
  ├── setup
  │   ├── human
  │   ├── beast
  │   └── transformed
  ├── adventure
  │   ├── human
  │   ├── beast
  │   └── transformed
  └── godTrial
ended
```

`rolling` 从领域状态中删除，改为 Vue 功能层的短命动画状态。

### 8.2 Process Manager 列表

| Process Manager | 消费的关键事件/信号 | 产生的主要事件 |
|---|---|---|
| `CharacterSetupProcess` | run started、setup option selected | 设定路线、安排性别/容貌/武魂/年龄/时期/势力 |
| `HumanProgressionProcess` | time advanced、level changed | 魂环、成长、遭遇、势力节点 |
| `BeastCultivationProcess` | cultivation changed、tribulation resolved | 成长、遭遇、雷劫、进化、化形 |
| `SoulRingProcess` | level threshold crossed、ring granted | 魂环与可选魂骨任务 |
| `GodTrialProcess` | god trial requested、exam resolved | 神位、考核、奖励、成神终局 |
| `StoryTimelineProcess` | timeline advanced、story choice resolved | 主线、势力线、杀戮之都、战争与战后事件 |
| `EndingProcess` | death、ascension、timeline completed | 唯一终局事件和队列清理 |

各 Process Manager 是纯函数：输入当前投影和新事件，输出后续领域事件或空数组。不通过全局 event bus 异步运行。

## 9. 内容包与编辑器

### 9.1 内容包

```text
src/content/
  manifest.json
  pools/
    setup/
    progression/
    story/
    beast/
    god-trial/
  entities/
    traits.json
    martial-souls.json
    domains.json
    soul-bones.json
    factions.json
  endings/
  schema/
  compiler/
```

`manifest.json` 至少包含 `schemaVersion`、`contentVersion`、内容文件列表和可选校验和。

内容编译器负责：

1. runtime schema 校验。
2. ID 唯一性和所有跨文件引用校验。
3. policy、fact、effect、signal 注册表校验。
4. 表达式深度、非法数字、空池、非正权重和无效结局校验。
5. 输出只读 `PresentationCatalog` 和 `MechanicsCatalog`。

### 9.2 结构化编辑器

编辑器按数据类型使用合适控件：

- 文案使用文本域。
- 启用状态使用开关，权重使用数字输入。
- 条件使用 all/any/not 组和可验证的 fact/operator/value 行。
- 效果使用 effect 类型菜单和对应的实体选择器、数字控件。
- policy 只允许从注册表选择，不允许输入 JavaScript。
- 预览同时显示当前资格判定、归一化概率和效果摘要。

纯叙事选项的 `effects` 为空数组。任何文案都不会自动推导效果。

### 9.3 本地覆盖 v3

- 新存储键：`douluo-wheel-overrides-v3`。
- 覆盖以 poolId + optionId 合并，不复制整个原始池。
- 导出文件包含 schemaVersion、contentVersion 和变更集。
- 覆盖在编译后必须重新走完整校验，不允许产生未注册 effect 或断引用。

- 所有产物来自同一 commit、contentVersion 和种子前缀。

#### 11.H.1 批次硬门禁

每个批次必须完整经过下列状态：

```text
not-started
  -> testing
  -> analyzing-and-reviewing
     -> no-defect-confirmed -> gate-verifying -> passed
     -> defect-confirmed -> fixing -> regression-verifying -> generation-invalidated
```

硬性规定：

1. **长程测试**：使用 `scripts/run-long-simulation.ts` 完整产生当批 20 份传记、20 份 JSON 归档、批次报告和 20 份审阅模板。任一产物缺失即本批失败。
2. **分析有无错误点**：必须读完 20 份传记，检查 20 份 JSON 的 audit/state/trace 摘要，完成 20 份审阅记录，并将每个可疑点判定为缺陷、内容质量问题或可接受行为。
3. **修复问题**：如有确认问题，当前批次不得 passed，必须修复、增加对应回归测试，使用原始 seed 完成定向复现/验证并通过 `npm run test:verify`。随后按 11.H.2 作废当前认证代次，新提交从新代次 batch-01 重新开始，不允许继续原代次下一批。如无确认问题，必须在批次报告中写明 `no-defect-confirmed`，不能留空。
4. **允许下一批**：只有自动审计为 20/20、逐份审阅为 20/20 passed、批次结论为 `no-defect-confirmed`、回归验证通过时，`gate.json.status` 才能变为 `passed`。否则脚本必须拒绝生成下一批。

`gate.json` 至少包含：

```json
{
  "batch": 1,
  "status": "passed",
  "commit": "<git-commit>",
  "contentVersion": "<content-version>",
  "automatedAudit": { "passed": 20, "total": 20 },
  "manualReview": { "passed": 20, "failed": 0, "pending": 0 },
  "analysisCompleted": true,
  "defectCount": 0,
  "fixOutcome": "no-defect-confirmed",
  "regressionVerified": true,
  "attempt": 1
}
```

脚本不能自动将 `manualReview` 标记为 passed；它只能验证已完成的逐份审阅记录。最终实施时使用两个显式动作：

```bash
# 只生成当批长模拟和 pending 审阅模板
SIMULATION_ACTION=run-batch SIMULATION_BATCH_INDEX=01 npm run simulate:long

# 在 20 份人工审阅完成后，校验证据、执行回归并尝试关闭门禁
SIMULATION_ACTION=verify-gate SIMULATION_BATCH_INDEX=01 npm run simulate:long
```

`verify-gate` 必须拒绝任何 pending/failed 审阅或确认缺陷，且不能改写人工结论。

#### 11.H.2 认证代次失效规则

- 任一批次发现需要修改代码或内容的缺陷时，当前 `certificationGeneration` 立即失效。
- 修复完成后使用新 commit/contentVersion 建立新代次，批次编号重置为 01。旧代次保留作为问题证据，但不计入最终 20 批。
- 因此，最终被计数的 batch-01 至 batch-20 必然来自同一 commit 和 contentVersion。无缺陷批次完成“长程测试 → 错误分析 → 无缺陷确认 → 门禁验证”；发现缺陷时完成“长程测试 → 错误分析 → 修复 → 回归验证 → 代次作废”。

每一份模拟必须执行以下审阅，不允许抽样：

1. 逐行阅读完整 `.txt` 传记，不只阅读头部摘要。
2. 检查对应 `.json` 的 audit、最终 state、EventBatch/trace 数量和结局，并与传记交叉核对。
3. 检查身份和路线：人类/魂兽/化形称谓、年龄、时间坐标、武魂/本体是否一致。
4. 检查成长与资源：等级/修为、魂环、魂骨、称号、领域、神考和神位的获得、移除、数量与前置条件。
5. 检查因果与时序：任务顺序、剧情分支、时间跳跃、战斗结果和后续奖励不自相矛盾。
6. 检查终局：死亡、存活、成神或时代终章与最终 state 一致，终局后无新事件。
7. 检查传记质量：无内部 ID/flag 泄露、无重复段落、无空栏目、无机制文本被当成实体、无明显语义矛盾。
8. 在对应 `reviews/<sample>.md` 中记录每项结论、发现、相关回合和最终 `passed/failed`。

批次审阅流程：

1. 按 batch-01 到 batch-20 顺序处理，每批完成 20 份审阅后更新 `batch-review.md`。
2. 批次报告必须记录 20 份产物存在性、自动审计、人工审阅、问题编号和审阅完整率。
3. 发现问题时，在 `问题排查/` 或归档内的 findings 目录记录最小复现 seed、回合、根因和回归测试 ID。
4. 任一确认缺陷都按 11.H.2 使当前认证代次失效。修复、增加回归测试并通过全套自动化后，必须从新目录的 batch-01 重新开始，不混用修复前后产物。

最终退出门禁：

- V03-SIM-001、V03-SIM-002、V03-BIO-001 至 V03-BIO-003、V03-CERT-001 全部通过。
- 400/400 份产物完整，400/400 自动审计通过，400/400 逐份审阅为 passed，0 pending，0 failed。
- `audit-report.md`、`manual-review-report.md`、`manifest.json`、20 份批次报告和 400 份审阅记录数量一致。
- 本认证之后未再修改代码或内容；如有修改，认证自动失效并重跑。

新命令只封装真实的 TypeScript/Vitest 入口，不建立第二套并行测试系统。`npm run test:verify` 在阶段 G 将包含内容校验。

### 13.3 最终长模拟命令契约

export SIMULATION_BATCH_SIZE=20

实施时还必须显式设置 `SIMULATION_ACTION=run-batch | verify-gate`，具体动作契约见 11.H.1。当 batch-01 的 `gate.json.status` 尚未达到 `passed` 时，任何 batch-02 的 `run-batch` 调用必须失败并给出可操作的原因。不允许使用 shell 循环、并行调用或脚本内自动进入下一批。

## 14. 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| 2,000+ 选项的语义转换工作量被低估 | 高 | 生成草案 + 分域审计 + 待判定清单归零，将阶段 F 的全量收口作为独立里程碑 |
| 通用规则模型演化为难维护 DSL | 高 | 只保留有限 Predicate/Expression；复杂规则使用注册纯函数 policy |
| Process Manager 互相触发产生循环 | 高 | 稳定步数上限、事件链诊断、批次取消和循环 fixture |
| 事件模型过度粒度化，代码量膨胀 | 中 | 事件只记录业务事实，不记录 UI 细节或每次中间计算 |
| XState 与领域状态形成两套真相 | 高 | 只持久化 DomainEvent，Statechart 作为同一 GameState 的生命周期投影，禁止额外 Vue 业务状态 |
| 转盘动画与立即提交导致用户感知错位 | 中 | UI 使用回合收据锁定交互，动画完成后再显示已投影结果 |
| 旧存档和覆盖失效引发用户困惑 | 中 | 使用全新 storage key；发布说明明确标注不兼容；不静默读取旧数据 |
| 重构期同时修改 v0.2 产生合并冲突 | 中 | 以稳定基线分支开始，按纵向切片合并，不在同一阶段混入内容扩写 |
| 400 份传记人工审阅出现疲劳、遗漏或抽样替代全查 | 高 | 每批限定 20 份，一份一记录，索引校验 20/20 后才能放行 |
| 脚本连续生成多批，绕过“测试 → 分析 → 修复” | 高 | 每次调用只生成一批，强制校验前批 `gate.json`，禁止自动递增 |
| 后续批次发现缺陷导致早期证据与新版本混用 | 高 | 任一代码/内容修复使当前代次失效，冻结新版本后从 batch-01 重启 |

## 15. 回滚与发布策略

### 15.1 开发期

- v0.3 内核在新模块中按纵向切片搭建，主 UI 在阶段 F 之前仍使用当前内核。
- 不在生产 UI 同时维护两个可交互的业务状态源。
- 纵向切片通过独立 fixture 页或测试入口验证，不向普通用户暴露半完成流程。

### 15.2 切换期

- 切换前为当前可部署版本建立 v0.2 标签或发布快照。
- v0.3 使用独立 storage key，因此可通过重新部署 v0.2 完成代码回滚，不需要逆向数据迁移。
- 阶段 F 完成后一次性切换主 UI，稳定窗口内不继续向旧内核同步新功能。

### 15.3 失败判定

出现以下任一情况时不可发布：

- 任一 P0 内容校验、确定性、原子性或主旅程用例失败。
- 仍有内核代码根据展示文本判断业务。
- 内容审计仍有“待人工判定”选项。
- 仍有未注册 effect/policy/signal 被静默忽略。
- 长模拟产生空池、非法状态、无法结算回合或事件循环。
- 任一批未完成“长程测试 → 错误分析 → 修复并重跑/无缺陷确认”即进入下一批。
- 任一传记或 JSON 归档缺失、审阅未完成，或人工状态为 pending/failed。
- 最终 20 批证据混用不同 commit、contentVersion 或 certificationGeneration。

## 17. Definition of Done

v0.3 只有在同时满足以下条件时才算完成：

1. 生产运行时不存在选项文本、池名称或剧情编号解析。
2. 所有机制数据、实体和引用通过 v0.3 内容 schema 与语义校验。
3. 人类、魂兽、化形、魂环、神考、剧情、死亡和终局不再由单一大函数编排。
4. 抽取、效果、流程反应和 RNG 变化作为一个 EventBatch 原子提交。
5. 回退、刷新恢复、传记和模拟均由相同事件日志驱动。
6. 编辑器可显式编辑并校验条件、权重和效果，改文案不改变机制。
7. `npm run test:verify` 通过，Playwright P0/P1 主旅程有证据。
8. `scripts/run-long-simulation.ts` 在唯一候选版本上完成 20 批 × 20 次，每批都先完成长程测试、错误分析和修复并重跑/无缺陷确认后才进入下一批。
9. 400/400 份 `.txt` 传记全文逐份审阅，400/400 份 `.json` 完成交叉核对，自动审计与人工审阅均为 passed。
10. `docs/test/v0.3-test-cases.json` 中所有 P0 用例有自动化实现和证据。
11. `README.md`、`docs/state-machine.md`、v0.3 自动化文档和实施记录反映当前事实。
12. 旧内核、旧 wheels.json、旧本地存储读取和所有运行时文本 fallback 已删除。

## 18. 已确定的架构决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 流程模型 | Statechart + Process Manager | 区分生命周期与跨回合领域编排 |
| 状态更新 | Domain Event + pure reducer | 可审计、可回放、可原子验证 |
| 内容条件 | 有限 Predicate/Expression | 支持组合，但不把 JSON 做成通用脚本 |
| 复杂规则 | 注册纯函数 policy | 保留 TypeScript 类型、测试和领域语义 |
| 流程触发 | 结构化 signal | 内容只表达意图，Process Manager 决定后续任务 |
| 存档 | EventBatch[] + contentVersion | 确定性恢复，明确拒绝不兼容内容 |
| 回退 | 按 turnId 截断并重放 | 与转盘结算原子边界一致 |
| 文案 | 独立 PresentationCatalog | 从类型层面阻止内核解析文本 |
| 兼容策略 | 不迁移 v0.2 数据 | 避免新内核继承旧语义和持久化包袱 |
| 长模拟执行 | 单批串行硬门禁 | 强制每批完成测试、分析和修复/确认后才继续 |
| 传记认证 | 400 份全查，不抽样 | 自动审计不能代替语义、因果和叙事质量检查 |
| 认证版本 | 修复即作废当前代次并从 batch-01 重启 | 防止混用修复前后证据 |
